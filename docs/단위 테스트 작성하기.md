# 단위 테스트 작성하기

## 모듈 모킹

모듈 모킹은 실제 모듈/객체와 동일한 동작을 하는 모의(mock) 모듈/객체를 만들어 실제를 대체하는 방식이다.

### 모킹의 장점과 단점

- **장점**: 외부 모듈과의 의존성을 제거하고 필요한 부분만 독립적으로 검증 가능
- **단점**:
  - 실제 모듈과 완전히 동일한 모의 객체를 구현하는 데 큰 비용이 소요
  - 모의 객체를 남용하면 테스트 신뢰성이 떨어질 수 있음

### 모킹을 활용하는 방법

- `vi.mock()`을 사용해 특정 모듈을 모킹할 수 있음
- 외부 모듈의 검증은 배제하고, 테스트 대상 컴포넌트를 독립적으로 검증 가능
- 단, 외부 모듈은 별도로 검증되어야 함

### 모킹 초기화

테스트의 독립성과 안정성을 보장하기 위해, 각 테스트가 끝난 뒤 모킹을 초기화해야 한다.

- **사용할 함수**:
  - `resetAllMocks`
  - `clearAllMocks`
  - `restoreAllMocks`
- **활용 위치**: `teardown` 단계에서 초기화 실행

## 훅(Hook) 테스트 작성하기

### 리액트 훅 테스트의 기본 원칙

- 리액트 훅은 반드시 리액트 컴포넌트 내에서 호출되어야 정상적으로 실행된다.
- 따라서 `renderHook` 함수를 활용해 테스트해야 한다.

### `renderHook` 함수 주요 기능

- **`result`**: 훅의 호출 결과를 반환, `result.current`를 통해 최신 상태를 참조 가능
- **`rerender`**: 새로운 인자를 전달하여 훅의 상태를 갱신

---

### act 함수의 역할

`act` 함수는 렌더링과 업데이트를 실제 앱의 동작과 유사하게 처리할 수 있도록 돕는 도구이다.

#### 예시(잘못된 코드)

```javascript
it('toggleIsModalOpened()를 호출하면 isModalOpened 상태가 변경된다.', () => {
  const { result } = renderHook(useConfirmModal);

  result.current.toggleIsModalOpened();

  expect(result.current.isModalOpened).toBe(true);
});
```

#### act 함수 사용 이유

- `act`는 상호 작용(렌더링, 이펙트 등)을 그룹화하고 실행하여, 가상의 DOM(jsdom)에 상태 변화가 올바르게 반영되도록 보장한다.
- 컴포넌트의 렌더링 이후 상태 업데이트 결과를 검증할 때 사용한다.

#### 예시 (올바른 코드)

```javascript
import { act, renderHook } from '@testing-library/react';

it('toggleIsModalOpened()를 호출하면 isModalOpened 상태가 변경된다.', () => {
  const { result } = renderHook(useConfirmModal);

  act(() => {
    result.current.toggleIsModalOpened();
  });

  expect(result.current.isModalOpened).toBe(true);
});
```

## 기타 유용한 함수

### setSystemTime

- 시간에 의존하는 테스트에서 시간을 고정하면 테스트 결과의 일관성을 유지할 수 있다.
- `setSystemTime`을 사용해 테스트 시점을 고정한다.

### fireEvent

- DOM 이벤트를 시뮬레이션하기 위한 도구이다.
- 내장되어 있기 때문에 별도의 설치 없이 사용 가능
- 단순하게 해당 이벤트만 디스패치
  - 클릭 이벤트에 대한 제어, 버블링, 캡처링만 가능
  - 실제로 사용자가 사용하는 것과 다르게 동작함
  - ex) fileEvent('click') 하더라도 focus 같은 이벤트는 발생하지 않음

### userEvent

- `fireEvent`보다 실제 사용자 동작을 더 잘 시뮬레이션할 수 있다.
- **특징**:
  - `disabled` 상태나 `display` 속성까지 고려하여 신뢰성 높은 테스트 제공
  - 예: 비활성화된 입력 필드에는 입력이 불가

> 단, 일부 이벤트(e.g., scroll)를 지원하지 않을 때는 `fireEvent`도 고려 가능

## 단위 테스트의 한계

### 단위 테스트의 적합한 경우

- 공통 컴포넌트, 커스텀 훅, 공통 유틸처럼 다른 모듈에 대한 의존성이 거의 없을 때
- 모듈 자체만으로 독립적인 역할을 수행할 때

### 단위 테스트의 한계점

- 여러 모듈이 조합될 때 발생하는 이슈를 발견하지 못함
- 앱의 전체적인 동작이 비즈니스 요구사항에 맞는지 보장하지 못함

> **해결 방안**:  
> 단위 테스트에서 검증하지 못하는 부분은 통합 테스트, E2E 테스트, 시각적 테스트 등을 통해 보완해야 한다.

---

## 통합 테스트의 역할

통합 테스트는 여러 모듈이 조합될 때 발생하는 상호 작용과 비즈니스 로직을 검증한다.

- **장점**: 비즈니스 로직 기준으로 여러 컴포넌트 간의 상호 작용을 한 번에 검증 가능
- **활용 사례**: 다양한 모듈이 함께 동작하는 시나리오 검증
